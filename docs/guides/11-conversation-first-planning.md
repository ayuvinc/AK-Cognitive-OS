# Guide 11 — Conversation-First Planning

## Why Conversation-First

AI agents are good at drafting, structuring, and summarizing. They are bad at knowing what the user actually wants. When an AI creates planning documents without structured user input, it fills gaps with plausible-sounding but unverified assumptions. This creates fake certainty — documents that look authoritative but are grounded in inference, not intent.

**The rule:** Core planning documents are conversation-derived artifacts. The AI must begin with a structured conversation, summarize answers, ask for confirmation, then create documents. It must not infer the whole project and start writing planning docs immediately.

---

## Discovery Conversation — 8 Questions

Before creating any planning artifacts, ask the user these questions. Ask one at a time. Wait for each answer before proceeding.

1. **Who is the primary user?** — Role, context, technical level. Not "everyone" — the specific person whose problem you're solving first.
2. **What exact problem is being solved?** — Be concrete. Not "they need a better tool" but "they spend 3 hours per week manually copying data between spreadsheets."
3. **What is the core user journey?** — The main thing the user does, step by step. One flow, not five.
4. **What are the top 3-5 must-have features?** — Features that absolutely must ship. Without these, the product has no value.
5. **What is out of scope?** — Things that are adjacent but explicitly excluded. This prevents scope creep.
6. **What is the delivery target?** — Demo, pilot, or production? This changes everything about quality, testing, and infrastructure.
7. **What are the constraints?** — Compliance, integrations, platforms, deadline, budget, team size.
8. **What defines success?** — One measurable outcome. Not "users like it" but "10 coordinators use it daily within 2 months."

After all 8 answers, summarize them back to the user and ask: "Is this accurate? Anything to change?"

---

## Artifact Creation Order

Create planning documents in this order. Each depends on the one before it.

```
1. problem-definition.md  ← from discovery conversation
2. scope-brief.md         ← from discovery conversation
3. assumptions.md         ← from both above + technical context
4. decision-log.md        ← ongoing, start with initial decisions
5. hld.md                 ← from architecture conversation (separate from discovery)
6. lld/<feature>.md       ← one per feature, only as needed
7. traceability-matrix.md ← after tasks are derived
8. release-truth.md       ← before demo or release
```

Do not skip steps. If a project is too small for full planning (see "When NOT to Plan" below), skip the whole flow — don't do it halfway.

## Metadata Header Rules

Every planning document starts with a metadata header:

```
---
Status: draft | confirmed | superseded
Source: user-confirmed | ai-inferred | code-observed | mixed
Last confirmed with user: YYYY-MM-DD
Owner: Product | Architect | Dev | QA
Open questions: 0
---
```

**Status meanings:**
- **draft** — initial version, not yet reviewed by user
- **confirmed** — user has reviewed and approved this content
- **superseded** — replaced by a newer version (keep for history)

**Source meanings:**
- **user-confirmed** — every statement was directly stated or approved by the user
- **ai-inferred** — AI derived this from context, conversation, or code patterns
- **code-observed** — generated by inspecting actual code and file structure
- **mixed** — combination of the above

**Rule:** A document with `Source: ai-inferred` must not be treated as authoritative until the user confirms it. Change Source to `user-confirmed` only after explicit user review.

---

## How to Summarize, Confirm, and Draft

1. **Listen first.** Ask the question. Let the user answer fully.
2. **Summarize.** Restate what you heard in structured form.
3. **Confirm.** Ask: "Is this accurate? Anything to add or change?"
4. **Draft.** Only after confirmation, write the document.
5. **Mark provenance.** Set `Source:` accurately. If you inferred anything beyond what the user said, mark it `mixed` and note which parts are inferred.

Never present an AI-drafted document as if the user wrote it. If the user hasn't confirmed a section, leave it blank or mark it `[UNCONFIRMED]`.

## When NOT to Do Full Planning

Not every task needs the full planning flow. Skip it when:

- **Single-file bug fix** — just fix it
- **Config change** — linting, CI, dependency update
- **Copy/text change** — no architectural impact
- **Trivial feature** — under 2 files, no business rules, no permissions
- **Spike / research** — you're exploring, not building

Use the full flow when:
- The feature touches 3+ files
- Business rules or permissions are involved
- Multiple people need to understand the design
- The feature will be demoed or released to users
- You're unsure about scope or approach

## Persona Ownership of Artifacts

Personas support artifact creation. They do not replace user input or create fake consensus.

| Artifact | Owner (decides) | Persona Support (drafts) |
|----------|----------------|-------------------------|
| problem-definition.md | Product (user/PM) | BA drafts from conversation |
| scope-brief.md | Product (user/PM) | BA + Architect draft |
| hld.md | Architect | Architect drafts, user confirms |
| lld/<feature>.md | Architect + Dev | Architect drafts, Dev refines |
| assumptions.md | All | BA maintains |
| decision-log.md | Product (user/PM) | All personas log decisions |
| release-truth.md | Product (user/PM) | QA + Architect contribute |
| traceability-matrix.md | Architect | Architect maintains |
| current-state.md | Architect | Architect drafts from code inspection |

**Key rule:** The "Owner" column is who decides if the content is correct. The "Persona Support" column is who helps write it. The user always has final authority.

## HLD Conversation

After problem-definition and scope-brief are confirmed, run a separate architecture conversation:

1. **System shape** — monolith, microservices, serverless, static site?
2. **Modules** — what are the major pieces and how do they relate?
3. **Data flow** — how does data enter, move through, and exit the system?
4. **Integrations** — what external systems does this connect to?
5. **Trust boundaries** — where does authentication and authorization happen?
6. **Deployment** — where does this run? How is it deployed?

Draft `docs/hld.md` and request user confirmation before proceeding to LLDs.

---

## LLD Conversation

For each feature that needs an LLD, ask targeted implementation questions:

1. What exactly should happen when the user does X?
2. What data is created, read, updated, or deleted?
3. Who has permission to do this? Who doesn't?
4. What happens when things go wrong? (validation failure, network error, race condition)
5. How do we know this feature works? (test scenarios)

Draft `docs/lld/<feature>.md` using the feature template. Mark unresolved questions explicitly.

---

## Common Mistakes

| Mistake | Fix |
|---------|-----|
| AI writes problem-definition without asking the user | Always start with discovery conversation |
| Document marked "confirmed" but user never reviewed it | Only set Status: confirmed after explicit user approval |
| HLD created before scope is clear | Confirm scope-brief.md first |
| LLD created for every feature at once | Create LLDs only for active + next 1-2 features |
| Tasks created without referencing design docs | Every task must trace to scope, HLD, and LLD |
| Release claims features as "real" without verification | Use release-truth.md with honest status |
