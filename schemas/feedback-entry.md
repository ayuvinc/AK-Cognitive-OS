# Feedback Entry Schema
# AK Cognitive OS — v4 Phase 2 Feedback Loop
# validation: markdown-contract-only | machine-validated

---

## Purpose

Defines the required and optional fields for memory entries written by Phase 2 feedback
sources (qa-run, risk-manager). These entries land in `memory/index.json` via
`mcp__ak-memory__write` and are the raw material for the Phase 3 Signal Engine.

This document is the authoritative reference for TASK-008, TASK-009, and TASK-010.

---

## Base Entry Fields

All memory entries receive these fields automatically from `memory_server.py`:

| Field | Type | Source |
|---|---|---|
| `entry_id` | string (MEM-NNN) | auto-generated by server |
| `session` | string | caller supplies |
| `timestamp_utc` | ISO-8601 string | auto-generated by server |
| `type` | string | caller supplies |
| `tags` | string[] | caller supplies |

Feedback entries must also supply the type-specific fields below.

---

## VALID_OUTCOMES

The `outcome` field is validated by `memory_server.py` against this set:

```
PASS | FAIL | PARTIAL | DEFERRED
```

Feedback entries use `PASS`, `FAIL`, and `PARTIAL` only. `DEFERRED` is reserved for
future use.

---

## Entry Type: "outcome"

Written by `qa-run` after each QA verdict.

### Required Fields

| Field | Type | Constraint |
|---|---|---|
| `type` | string | must be `"outcome"` |
| `task_id` | string | TASK-ID under review (e.g. `"TASK-042"`) — must not be empty |
| `persona` | string | must be `"QA"` |
| `outcome` | string | see outcome mapping below — must not be empty |
| `content` | string | one-line verdict summary ≤500 chars — task IDs and codes only |

### Outcome Mapping

| QA Verdict | outcome value |
|---|---|
| QA_APPROVED | `"PASS"` |
| QA_REJECTED | `"FAIL"` |

### Recommended Tags

```json
["qa", "verdict"]
```

### Example Call

```
mcp__ak-memory__write(
  type="outcome",
  outcome="PASS",
  task_id="TASK-042",
  persona="QA",
  session="26",
  tags=["qa", "verdict"],
  content="TASK-042 QA_APPROVED — all 6 ACs pass, build clean"
)
```

### Content Rules

- One sentence maximum
- Include: task ID, verdict code, AC pass count or failure summary
- Exclude: user data, log output, file contents, PII of any kind
- Cap: 500 chars (enforced by server — write will fail if exceeded)

## Entry Type: "decision"

Written by `risk-manager` after each risk assessment.

### Required Fields

| Field | Type | Constraint |
|---|---|---|
| `type` | string | must be `"decision"` |
| `task_id` | string | primary TASK-ID under assessment; `"cross-cutting"` if multi-task run |
| `persona` | string | must be `"risk-manager"` |
| `outcome` | string | see outcome mapping below — must not be empty |
| `content` | string | risk count summary ≤500 chars — counts and IDs only |

### Outcome Mapping

| Risk State | outcome value |
|---|---|
| No S0 or S1 risks open | `"PASS"` |
| S1 risks open, no S0 | `"PARTIAL"` |
| S0 risks open (any) | `"FAIL"` |

### Recommended Tags

```json
["risk", "assessment"]
```

### Example Call

```
mcp__ak-memory__write(
  type="decision",
  outcome="PARTIAL",
  task_id="TASK-007",
  persona="risk-manager",
  session="26",
  tags=["risk", "assessment"],
  content="4 risks reviewed, 1 new, S0: 0 open, S1: 1 open"
)
```

### Content Pattern

```
"N risks reviewed, N new, S0: N open, S1: N open"
```

- Use exact pattern for Signal Engine parseability in Phase 3
- Cap: 500 chars (enforced by server)
- Do not include risk descriptions, task descriptions, or any free-form text

### task_id Convention

| Scenario | task_id value |
|---|---|
| Single task under assessment | `"TASK-NNN"` |
| Multiple tasks or sprint-wide assessment | `"cross-cutting"` |
| Risk register maintenance (no active task) | `"cross-cutting"` |

## Write Failure Handling

Both `qa-run` and `risk-manager` must treat memory write failures as non-blocking:

1. Catch write failure (tool error, MCP unavailable, server down)
2. Add entry to `warnings[]` in the HANDOFF envelope
3. Set `memory_written: false` in `extra_fields`
4. Continue with verdict/assessment output unchanged — do NOT set status to FAIL or BLOCKED

The feedback layer is advisory in v4.0. A missing entry is recoverable; a blocked
workflow is not.
